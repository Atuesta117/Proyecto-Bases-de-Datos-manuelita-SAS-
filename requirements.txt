PROYECTO WEB - REQUERIMIENTOS FUNCIONALES DEL BACKEND

OBJETIVO
--------
Desarrollar el backend de una aplicación web para la gestión de una empresa utilizando PostgreSQL.

La aplicación debe implementar operaciones CRUD y cumplir todas las reglas de negocio descritas a continuación.

==========================================================
1. MÓDULO CLIENTES
==========================================================

CRUD completo.

CREATE
- Registrar cliente.
- Validar que el documento no exista previamente.
- Validar campos obligatorios.
- Guardar autorización Habeas Data.

READ
- Obtener listado de clientes.
- Obtener un cliente por ID.
- Buscar por nombre, documento o NIT.

UPDATE
- Permitir modificar toda la información excepto:
    * Tipo de documento
    * Número de documento

DELETE
- Solo eliminar si el cliente NO posee facturas asociadas.
- Si tiene facturas debe devolver un error.

==========================================================
2. MÓDULO PROVEEDORES
==========================================================

CRUD completo.

CREATE
Registrar proveedor con:

- Datos generales
- RUT
- Información bancaria
- Tipo de proveedor
- Tiempo promedio de entrega
- Contactos
- Condiciones de pago
- Calificación

READ
- Listar proveedores
- Obtener proveedor por ID
- Buscar por nombre o NIT

UPDATE
- Permitir modificar toda la información excepto el NIT.

DELETE
- Impedir eliminar si existen:
    - Productos asociados
    - Órdenes de compra asociadas

==========================================================
3. MÓDULO PRODUCTOS / INSUMOS
==========================================================

CRUD completo.

CREATE

Validaciones:

- Debe existir previamente un proveedor.
- No puede crearse un producto sin proveedor.

Campos mínimos:

- Código
- Nombre
- Stock actual
- Demanda diaria
- Precio
- IVA
- Proveedor

READ

Al consultar un producto debe calcular automáticamente:

dias_stock = stock_actual / demanda_diaria

NO guardar este valor en la base de datos.

También calcular automáticamente:

Si días_stock == 0

Estado:
AGOTADO

Acción:
Pedido inmediato

Si días_stock < 5

Estado:
CRÍTICO

Acción:
Pedido de emergencia

Si 5 <= días_stock <=15

Estado:
ALERTA

Acción:
Pedido normal

Si días_stock >15

Estado:
SEGURO

Acción:
Mantener monitoreo

UPDATE

Permitir modificar toda la información.

DELETE

No eliminar físicamente.

Implementar eliminación lógica:

activo = false

==========================================================
4. MÓDULO FACTURACIÓN
==========================================================

Registrar ventas.

No se requiere UPDATE ni DELETE.

Una factura debe contener:

Cabecera

- Cliente
- Fecha
- Hora
- Número consecutivo
- Prefijo

Detalle

- Uno o más productos
- Cantidad
- Valor unitario
- IVA por producto

Validaciones

Debe existir el cliente.

Debe existir el producto.

Cantidad mínima:

1

No permitir vender más unidades que las disponibles.

Al finalizar la factura:

Actualizar inmediatamente el stock.

Calcular automáticamente:

Subtotal

IVA

Total

Tarifas de IVA

19%

5%

0%

Excluido

Las tarifas deben obtenerse desde el producto.

==========================================================
5. MÓDULO ÓRDENES DE COMPRA
==========================================================

Registrar órdenes de pedido hacia proveedores.

No requiere UPDATE ni DELETE.

Validaciones

Debe existir el proveedor.

Debe existir el producto.

Cantidad mínima:

1

Campos:

Número de orden

Proveedor

Producto

Cantidad

Costo unitario

Fecha

Lugar de entrega

Estado

El estado inicial siempre debe ser:

Pendiente

==========================================================
6. CONSULTAS
==========================================================

Implementar endpoints para:

Clientes

GET /clientes

GET /clientes/{id}

Proveedores

GET /proveedores

GET /proveedores/{id}

Productos

GET /productos

GET /productos/{id}

Facturas

GET /facturas

GET /facturas/{id}

Órdenes

GET /ordenes

GET /ordenes/{id}

==========================================================
7. BÚSQUEDAS
==========================================================

Permitir filtros por:

Clientes

- nombre
- documento

Proveedores

- nombre
- NIT

Productos

- nombre
- proveedor

Facturas

- cliente
- fecha

Órdenes

- proveedor
- estado

==========================================================
8. REGLAS DE NEGOCIO
==========================================================

No registrar productos sin proveedor.

No registrar órdenes sin proveedor.

No registrar facturas sin cliente.

No vender productos inexistentes.

No vender productos sin stock.

Actualizar inventario inmediatamente después de una venta.

Calcular automáticamente:

- IVA
- subtotal
- total
- días de stock
- categoría de inventario
- acción recomendada

No almacenar días de stock.

==========================================================
9. ERRORES
==========================================================

Responder mensajes claros cuando:

Cliente inexistente.

Proveedor inexistente.

Producto inexistente.

Stock insuficiente.

Documento repetido.

NIT repetido.

Intento de eliminar cliente con facturas.

Intento de eliminar proveedor con órdenes.

==========================================================
10. ARQUITECTURA ESPERADA
==========================================================

Cada controlador debe contener únicamente la lógica HTTP.

La lógica de negocio debe estar separada en servicios.

El acceso a PostgreSQL debe estar separado en un repositorio o capa DAO.

Los controladores deben devolver respuestas JSON.

Usar códigos HTTP correctos:

200 OK

201 Created

400 Bad Request

404 Not Found

409 Conflict

500 Internal Server Error

==========================================================
11. OBJETIVO FINAL
==========================================================

El backend debe permitir administrar completamente:

Clientes

Proveedores

Productos

Facturas

Órdenes de compra

cumpliendo todas las reglas de integridad y de negocio definidas anteriormente.
